GUIA PROVISIONAL DE DESPLIEGUE EN AWS CON N8N
Proyecto: CAP Prenatal - CAP El Chal

ESTADO DE ESTA GUIA
-------------------

El repositorio no contiene todavia una configuracion de produccion. El archivo
docker-compose.yml es exclusivamente local: publica puertos, fija
NODE_ENV=development y no debe exponerse a Internet ni ejecutarse en un servidor
productivo.

No seguir versiones anteriores de esta guia que indicaban crear backend/.env y
levantar el Compose local en AWS. El archivo backend/.env no se copia a la imagen
y no sustituye la inyeccion segura de variables al contenedor.

El despliegue productivo queda bloqueado hasta un sprint posterior que incorpore
la topologia, TLS, redes privadas, usuarios no privilegiados y configuracion de
n8n apropiados para produccion.


1. Arquitectura futura minima
-----------------------------

  - Frontend detras de HTTPS.
  - Backend en red privada, sin publicar directamente el puerto de la API.
  - PostgreSQL en red privada y con credenciales propias del entorno.
  - n8n protegido y separado de la exposicion publica innecesaria.
  - Gestor de secretos o variables protegidas del proveedor.
  - Solo 80/443 publicos; SSH restringido a direcciones administradas.


2. Variables que debera inyectar el entorno
--------------------------------------------

Backend:

  NODE_ENV
  JWT_SECRET
  ACCESS_TOKEN_TTL_MINUTES
  SESSION_IDLE_TIMEOUT_MINUTES
  SESSION_WARNING_MINUTES
  SESSION_ABSOLUTE_HOURS
  SESSION_ACTIVITY_UPDATE_SECONDS
  FRONTEND_URL
  COOKIE_SAMESITE
  DB_SSL
  DB_SSL_REJECT_UNAUTHORIZED

Conexion PostgreSQL, una de estas dos modalidades:

  DATABASE_URL

o todas las siguientes:

  DB_HOST
  DB_PORT
  DB_NAME
  DB_USER
  DB_PASSWORD

Integraciones opcionales:

  AUTOMATION_SECRET

No escribir valores en esta guia, Compose, Dockerfile, comandos de shell, tickets
o logs. JWT, PostgreSQL y automatizacion deben usar valores aleatorios distintos.


3. Validacion del backend
-------------------------

El backend valida la configuracion antes de cargar rutas. En produccion rechaza:

  - NODE_ENV ausente o desconocido.
  - JWT_SECRET ausente, corto o con marcadores de ejemplo/desarrollo.
  - Configuracion PostgreSQL incompleta o predecible.
  - DB_SSL con un valor distinto de true/false.
  - FRONTEND_URL con HTTP, loopback, comodines, credenciales, rutas, query o
    fragmentos.
  - COOKIE_SAMESITE fuera de los valores admitidos.
  - AUTOMATION_SECRET inseguro cuando se configura.

Los errores identifican la variable invalida sin imprimir su valor.

Antes de iniciar la nueva version se debe ejecutar deliberadamente, desde backend:

  npm run db:migrate

El ejecutor aplica schema.sql, descubre en orden las migraciones versionadas
pendientes y registra su checksum en schema_migrations; esto incluye
007_auth_sessions.sql. No ejecutar el comando contra una base real sin respaldo,
ventana y validacion del entorno. Los JWT anteriores, que no contienen sid, dejan
de ser compatibles y los usuarios deben autenticarse otra vez. La limpieza manual
de sesiones antiguas puede ejecutarse posteriormente con npm run sessions:cleanup
desde backend; no es requisito para autenticar.

El access dura 10 minutos por defecto, la inactividad 15 minutos y la sesion
absoluta 8 horas. Este ultimo valor no restringe horarios laborales: al vencer,
el usuario puede iniciar una sesion nueva inmediatamente.

Un despliegue institucional interno puede usar un nombre DNS interno o una IP
privada en FRONTEND_URL, siempre mediante HTTPS. El certificado debe ser confiable
en todos los equipos que usen el sistema; se recomienda un nombre DNS interno en
lugar de una IP para facilitar certificados y mantenimiento. Permitir una direccion
privada concreta no equivale a permitir cualquier origen: CORS compara solamente
los origenes exactos configurados.


4. Cuenta inicial
-----------------

El seed ya no contiene cuentas ni contrasenas conocidas. Solo crea, si no existe,
una cuenta director definida mediante:

  SEED_DIRECTOR_NAME
  SEED_DIRECTOR_USERNAME
  SEED_DIRECTOR_PASSWORD

En produccion se requiere ademas SEED_CONFIRM_PRODUCTION con la confirmacion
deliberada documentada en docs/ROTACION_SECRETOS.md. Una cuenta existente no
recibe una contrasena nueva al volver a ejecutar el seed.


5. Rotacion obligatoria antes del futuro despliegue
---------------------------------------------------

Seguir docs/ROTACION_SECRETOS.md. Los secretos usados anteriormente estuvieron
presentes en el historial Git y deben rotarse aunque el repositorio sea privado.
Eliminar el archivo del ultimo commit no elimina sus versiones historicas.


6. n8n
------

Este sprint no cambia n8n. La configuracion funcional local sigue documentada en
docs/N8N.md. Antes de produccion se deberan definir por separado cifrado, retencion,
autenticacion, red y rotacion de AUTOMATION_SECRET.


7. Criterio de habilitacion futuro
----------------------------------

Antes de publicar el sistema se debera verificar, como minimo:

  - Imagenes y configuracion productivas revisadas.
  - Secretos inyectados desde un canal protegido.
  - JWT antiguos rechazados despues de la rotacion.
  - PostgreSQL, backend y n8n no expuestos directamente.
  - HTTPS y origen CORS productivo confirmados.
  - Migracion y seed ejecutados como tareas deliberadas, no al arrancar.
  - Pruebas y restauracion operativa aprobadas.

Fin de la guia provisional.
