GUIA DE TOPOLOGIA PRODUCTIVA DE EJEMPLO CON N8N
Proyecto: CAP Prenatal - CAP El Chal
Sprint: 5B.2B

ADVERTENCIA
-----------

docker-compose.production.example.yml es una configuracion revisable, no un
despliegue listo para produccion. No se ejecuto Docker, no se desplego en AWS,
no se configuraron credenciales reales y no se enviaron correos. El workflow
n8n/workflows/proximas-citas-v1.json existe versionado, pero esta inactivo y no
se importo. No hubo cambios de base de datos o archivos .env reales.


1. TOPOLOGIA
------------

Internet/navegador
        |
        v
proxy Nginx :80/:443 -- app_internal --> backend :3001
                                             |
                                             +-- data_internal --> PostgreSQL
                                             |
n8n -- automation_internal ------------------+

Redes:

- proxy_public: proxy.
- app_internal 172.30.10.0/24: proxy y backend.
- data_internal 172.30.20.0/24: backend y PostgreSQL.
- automation_internal 172.30.30.0/24: n8n y backend.

Las tres redes internas usan internal=true. n8n no comparte red con
PostgreSQL. Verificar antes del despliegue que estas subredes no colisionen con
VPC, VPN o redes Docker existentes.


2. PUERTOS
----------

Produccion:

- proxy: publica 80. TLS debe terminar en el proxy o balanceador administrado.
- backend: solo expose 3001; sin ports.
- PostgreSQL: solo expose 5432; sin ports.
- n8n: solo expose 5678; sin ports.

La UI n8n se administra solo mediante VPN, red administrativa o tunel
autenticado, con HTTPS si cruza red. Nunca publicar 5678 directamente.
El proceso n8n usa HTTP dentro de la red Docker; el HTTPS administrativo debe
terminar en el componente privado de VPN/tunel/proxy que se agregue al entorno.
No abrir ese acceso hasta verificar la cuenta propietaria y la autenticacion de
n8n; una UI sin autenticacion no cumple este diseño.


3. PROXY Y ENDPOINT INTERNO
---------------------------

frontend/nginx.conf bloquea antes del /api/ general:

  /api/automatizaciones/

La regla devuelve 404 JSON uniforme, es case-insensitive y trabaja sobre la URI
normalizada con merge_slashes activado. Las rutas humanas restantes continúan
enviandose a backend:3001.

n8n omite el proxy publico y usa:

  http://backend:3001/api/automatizaciones/v1/proximas-citas


4. TRUST PROXY
--------------

TRUSTED_PROXY_CIDRS=172.30.10.0/24 autoriza exclusivamente la subred del proxy
humano. Express usa una funcion CIDR estricta y nunca trust proxy=true.

El endpoint M2M no usa req.ip ni X-Forwarded-For: valida siempre la direccion
del socket directo de automation_internal contra:

  N8N_ALLOWED_CIDRS=172.30.30.0/24


5. SEPARACION DE SECRETOS
-------------------------

Backend recibe:

- JWT_SECRET.
- POSTGRES_PASSWORD o credenciales DB equivalentes.
- N8N_API_KEY_HASH_CURRENT/NEXT.

n8n recibe:

- N8N_ENCRYPTION_KEY.
- URLs y configuracion propia.
- Politica de retencion.
- Limite de concurrencia productiva 1.

n8n no recibe credenciales PostgreSQL clinicas, JWT, CSRF, sesiones, SMTP del
backend, seed ni hashes M2M. La API key original queda solamente como credencial
Header Auth en n8n.

deploy/.env.example es inventario sin secretos. En produccion, inyectar valores
mediante gestor de secretos, archivos protegidos o entorno administrado. No
guardar secretos en Git, tickets, logs o el Compose.


6. VERSION
----------

n8n queda fijado en 2.26.4, coincidente con el lockfile. No se afirma que sea la
version mas reciente. Antes de actualizar, revisar notas de version, respaldar,
probar restauracion y compatibilidad, actualizar todos los puntos fijados y
evitar saltos automaticos de major.


7. CIFRADO, PERSISTENCIA Y BACKUP
---------------------------------

N8N_ENCRYPTION_KEY es obligatoria y debe ser estable. Perderla o regenerarla
puede volver indescifrables las credenciales existentes.

El volumen cap_prenatal_production_n8n_data es persistente. El contenedor usa
UID/GID 1000:1000, no-new-privileges y exige permisos de settings. Definir en el
host permisos, propietario y cifrado de respaldo compatibles con la plataforma.

Respaldar por separado:

- volumen n8n;
- N8N_ENCRYPTION_KEY en gestor de secretos;
- procedimiento de restauracion verificado.

No colocar la unica copia de la clave dentro del mismo backup del volumen.


8. RETENCION
------------

Valores iniciales:

- EXECUTIONS_DATA_PRUNE=true.
- EXECUTIONS_DATA_MAX_AGE=168 horas.
- EXECUTIONS_DATA_PRUNE_MAX_COUNT=1000.
- SAVE_ON_SUCCESS=none.
- SAVE_ON_ERROR=none.
- SAVE_MANUAL_EXECUTIONS=false.
- SAVE_ON_PROGRESS=false.
- N8N_CONCURRENCY_PRODUCTION_LIMIT=1.

El workflow v1 tambien deshabilita guardado de exito, error, manual y progreso.
No devuelve headers, descarta inmediatamente errores y respuestas despues de
validar y solo produce codigos, cantidades, rangos y hashes. No contiene
pinData.


9. HEALTH CHECKS
----------------

- PostgreSQL: pg_isready.
- Backend: GET /api/health.
- n8n: GET interno /healthz.
- Proxy: GET local /healthz.

No usan API keys, login, consultas clinicas ni el endpoint de citas.


10. WORKFLOW VERSIONADO
-----------------------

El workflow se ejecutara todos los dias a las 06:00 America/Guatemala y consulta
unicamente manana con offset_days=1/window_days=1. Valida el contrato v1, no
recibe pacientes individuales, no usa PostgreSQL y produce un solo correo de
texto agregado. total=0 finaliza sin SMTP.

La autenticacion usa una credencial Header Auth:

  X-CAP-Automation-Key

La key original queda solo en n8n y no se incluye en el JSON o entorno del
backend. Destinatario, remitente y alias son variables de proyecto n8n; SMTP es
una credencial separada. Los placeholders example.invalid bloquean el envio.

Static data usa SHA-256 de v1|from|to|alias, guarda solo hash/timestamp y marca
despues del envio confirmado. Es mejor esfuerzo, no una garantia transaccional.
N8N_CONCURRENCY_PRODUCTION_LIMIT=1 y una unica copia activa reducen la carrera;
revisar el impacto global antes de alojar otros workflows en esta instancia.


11. SALIDA SMTP PENDIENTE
-------------------------

No agregar egress general. Evaluar, en este orden:

A. Relay SMTP institucional unido a automation_internal.
B. Red de egress dedicada con firewall limitado al proveedor aprobado.
C. Servicio institucional accesible desde red privada.

No conectar n8n a data_internal, PostgreSQL o una red publica sin controles.
No publicar 5678. El workflow solo permite un reintento SMTP ante rechazo
transitorio claramente previo a aceptacion; un timeout ambiguo no se reintenta.
