1. VERSION FIJADA
-----------------

La dependencia local y la imagen Docker usan n8n 2.26.4. No se usa latest,
rango semver ni descarga implicita con npx. Esta version coincide con el
package-lock actual; no se afirma que sea la mas reciente.

Antes de actualizar:

1. Revisar notas de version y cambios incompatibles.
2. Respaldar el directorio/volumen n8n y probar restauracion.
3. Validar nodos y credenciales en un entorno aislado.
4. Actualizar package.json, lock y ambos Compose en el mismo cambio.
5. Ejecutar las pruebas de infraestructura.


2. CONFIGURACION DEDICADA
-------------------------

Copiar:

  n8n/.env.example

a:

  n8n/.env

El archivo real queda ignorado por Git. Completar N8N_ENCRYPTION_KEY con al
menos 32 caracteres aleatorios. La clave debe ser estable: regenerarla despues
de crear credenciales puede impedir descifrarlas.

No agregar a n8n/.env:

- DB_PASSWORD, DATABASE_URL o POSTGRES_* clinicos.
- JWT, CSRF o SESSION secrets.
- SMTP del backend.
- SEED_*.
- N8N_API_KEY_HASH_CURRENT/NEXT.
- AUTOMATION_SECRET.
- La API key M2M original.

La API key se almacenara como credencial Header Auth dentro de n8n, no como
variable compartida. El workflow versionado no contiene la key ni un ID local
de credencial.


3. INICIO MEDIANTE POWERSHELL
-----------------------------

Desde la raiz:

  npm run n8n:local

scripts/start-n8n-local.ps1:

- lee unicamente n8n/.env;
- acepta una lista cerrada de variables;
- elimina variables clinicas heredadas del proceso;
- exige N8N_ENCRYPTION_KEY;
- exige America/Guatemala;
- exige N8N_LISTEN_ADDRESS en loopback;
- valida las URLs dedicadas;
- exige n8n local 2.26.4;
- no usa backend/.env;
- no usa npx;
- no inicia si falta configuracion o instalacion local.

Instalar dependencias desde el lockfile antes del primer uso:

  npm install


4. INICIO MEDIANTE DOCKER LOCAL
-------------------------------

docker-compose.yml sigue siendo exclusivamente local. Requiere los secretos
vacios indicados por .env.example y publica solo en loopback:

  Frontend:   127.0.0.1:8080
  Backend:    127.0.0.1:3001
  PostgreSQL: 127.0.0.1:5432
  n8n:        127.0.0.1:5678

El contenedor n8n recibe solo variables declaradas en su bloque environment.
No recibe secretos del backend aunque Compose lea el archivo raiz para
interpolacion.

Las redes locales estan separadas:

- app_internal: frontend y backend.
- data_internal: backend y PostgreSQL.
- automation_internal: n8n y backend.

n8n no comparte la red de PostgreSQL. Su URL privada futura es:

  http://backend:3001/api/automatizaciones/v1/proximas-citas

No ejecutar Docker como parte de este sprint.


5. RETENCION LOCAL
------------------

La instancia local conserva solo errores y ejecuciones manuales sinteticas, con
poda habilitada, maximo 24 horas y 100 ejecuciones. El workflow v1 aplica una
politica aun mas estricta y deshabilita guardado de exito, error, ejecuciones
manuales y progreso. No usar datos reales de pacientes.


6. ACCESO
---------

La UI local se liga a 127.0.0.1. No cambiarla a 0.0.0.0. Si se necesita acceso
desde otra computadora, usar un tunel autenticado y HTTPS; no publicar 5678 en
la red.

8. IMPORTAR EL WORKFLOW
-----------------------

1. Abrir la UI local solo en 127.0.0.1:5678.
2. Importar n8n/workflows/proximas-citas-v1.json.
3. Confirmar que el nombre termina en v1 y aparece inactivo.
4. Confirmar horario diario 06:00 y America/Guatemala.
5. No publicar/activar.
6. Confirmar que no exista pinData.

El flujo consulta manana con offset_days=1/window_days=1, valida estrictamente
la respuesta agregada, calcula SHA-256, comprueba static data, termina sin
correo cuando total=0 y solo marca la clave despues de un envio confirmado.


9. CREDENCIAL HEADER AUTH
-------------------------

Crear una credencial generica Header Auth:

  Header name: X-CAP-Automation-Key
  Header value: <key M2M guardada por canal seguro>

Asignarla a los tres nodos "Consultar backend - intento N". No pegar la key en
la URL, query, expresiones, notas o variables. La key original queda solo en la
credencial n8n; el backend recibe sus hashes CURRENT/NEXT.


10. CORREO Y PARAMETROS OPERATIVOS
----------------------------------

Crear una credencial SMTP sintetica o aprobada y asignarla a ambos nodos
"Enviar resumen - intento N". No exportar host, usuario, password, token o IDs
locales de credenciales.

Crear variables de proyecto n8n:

  CAP_NOTIFICATION_RECIPIENT
  CAP_NOTIFICATION_FROM
  CAP_NOTIFICATION_RECIPIENT_ALIAS

El alias puede ser responsable_salud_reproductiva y no debe contener el correo.
Los placeholders responsable@example.invalid y cap-prenatal@example.invalid
son deliberadamente no funcionales. CAP_SYSTEM_BASE_URL debe ser HTTPS en una
ejecucion programada. javascript:, data: y file: se rechazan.


11. PRUEBA SINTETICA
--------------------

Antes de abrir n8n:

  node --test backend/test/n8nWorkflow.test.js

La suite ejecuta el validador con contratos sinteticos, incluidos total=0,
campos adicionales, suma incorrecta y campos prohibidos. No requiere Docker,
n8n, SMTP o PostgreSQL.

Para una prueba manual posterior, usar un mock local con la ruta exacta
/api/automatizaciones/v1/proximas-citas y una respuesta sintetica total=0.
Configurar temporalmente CAP_BACKEND_AUTOMATION_URL hacia ese mock. Verificar
resultado no_results y que ningun nodo SMTP se ejecute. No usar capturas o
datos reales.


12. PRIVACIDAD, IDEMPOTENCIA Y CONCURRENCIA
-------------------------------------------

El workflow no recibe pacientes individuales y no conserva body completo,
headers o respuesta SMTP en salidas. Produccion tampoco guarda ejecuciones.
Los logs permitidos se limitan a codigos, cantidades, rangos y hashes.

Static data conserva solo hasta 90 hashes/timestamps por 45 dias. Es de mejor
esfuerzo y no es una garantia transaccional. La concurrencia operativa debe
quedar en 1 y solo puede existir una copia activa; la configuracion productiva
de este sprint usa N8N_CONCURRENCY_PRODUCTION_LIMIT=1.


13. ACTIVACION Y EGRESS PENDIENTES
----------------------------------

No activar sin autorizacion institucional, proveedor/destinatario aprobados,
concurrencia confirmada y egress restringido. Opciones:

A. Relay SMTP institucional dentro de automation_internal.
B. Red de egress dedicada y firewall limitado al proveedor SMTP aprobado.
C. Servicio institucional accesible desde red privada.

Se prefiere A o B con allowlist estricta. No conectar n8n a data_internal o
PostgreSQL, no publicar 5678 y no agregar una red publica general.
